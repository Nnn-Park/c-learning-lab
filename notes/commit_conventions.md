- 앞으로 이 레포를 “연구실”처럼 쓰려면 어떤 규칙이 필요할까?

---

# 🧭 Commit Convention for C-Learning-Lab

이 문서는 **c-learning-lab** 레포에서 사용하는  
커밋 메시지 규칙을 정리한 “연구실 운영 문서”다.

이 규칙은 다음 목적을 가진다:

- 실험(experiment), 연구노트(note), 정리(docs) 흐름을 명확히 한다  
- 나중에 로그를 되돌아볼 때 “그 날 내가 어떤 공부를 했는지” 쉽게 파악  
- 브랜치 전략과 자연스럽게 통합  
- 장기적으로 정돈된 연구 레포를 유지하기 위함

---

# 🏷 Prefix 규칙 (영어로 통일)

| Prefix | 의미 | 사용 상황 |
|--------|------|------------|
| **feat:** | 새로운 기능/파일 추가 | 새로운 C 예제, 실험, 기능 코드 |
| **fix:** | 버그 수정 | 코드 오류, 출력 형식 문제, 오타 수정 |
| **docs:** | 문서/정리 | 개념 정리, README, 이론 설명 md |
| **note:** | 연구 로그 | 내가 오늘 배운 점, 실험 생각, 개인 메모 |
| **exp:** | 실험 코드 | 특정 개념 테스트용 코드 작성 |
| **refactor:** | 구조 개선 | 폴더 이동, 파일 이름 변경, 구조 리팩터링 |
| **chore:** | 기타 관리 | gitignore 수정, 설정 파일 변경 |
| **test:** | 테스트 코드 | 테스트용 C 코드 작성 |
| **style:** | 코드 스타일 | 포맷팅, 공백, 들여쓰기, 논리 변경 없음 |
